## 리팩토링은 왜 해야 하나?

- 소프트웨어 설계 개선

  - 코드 설계가 깔끔하지 않으면 기능이 같은 코드가 중복되어 코드량이 많아짐

  - 이런 중복코드를 없애면 나중에 코드 수정시 절감하게 마련
</br>

- 다른 개발자가 소프트웨어를 이해하기 더 쉬워짐

  - 한 시간 정도면 이해하고도 남을 코드를 

    다른 개발자가 신속히 이해하지 못해 

    한 부분을 수정하는 데 일주일 이상 걸린다면 큰 문제

  - 코드를 이해하기 쉬우면 버그를 찾기 쉽고, 프로그래밍 속도가 빨라짐
</br>

## 리팩토링은 어떨 때 필요한가?

- 같은 작업을 3번째 반복하게 됐을 때
- 기능 추가, 버그 수정, 코드 검수 작업시

  (코드를 이해하기 쉬우면 위 작업이 수월해짐)
</br>

## 리팩토링하면 안 되는 상황

- 기존 코드가 정상 동작을 하지 않을 때
  - 코드는 반드시 대부분 제대로 돌아가는 것이 우선이고, 리팩토링은 나중 일
  
- 프로젝트 남기가 임박할 때
  - 납기가 가까운 시점에 리팩토링해봤자 그로 인한 생산성은
  
    납기가 지난 후에야 가시화될 테니 쓸데 없음
